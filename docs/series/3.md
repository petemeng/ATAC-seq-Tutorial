## ATAC-seq 为什么用 Nextera 接头？

在进入质控之前，有一个重要的背景知识你需要了解。

大多数 NGS 文库（RNA-seq、WGS 等）使用 **TruSeq 接头**进行文库构建。但 ATAC-seq 不同——它使用的是 **Nextera 接头**。原因很简单：**Tn5 转座酶上预载的就是 Nextera 接头序列**。

两种接头的序列不同：

| 接头类型 | Read 1 端序列 |
|----------|--------------|
| **Nextera (ATAC-seq)** | `CTGTCTCTTATACACATCT` |
| TruSeq (RNA-seq 等) | `AGATCGGAAGAGCACACGTCTGAACTCCAGTCA` |

💡 **为什么这一点很重要？**
> 如果你用 TruSeq 接头序列去做接头去除，ATAC-seq 数据中的 Nextera 接头会被完全漏掉，导致大量含接头的 reads 进入比对步骤。好消息是：**`fastp` 的 `--detect_adapter_for_pe` 功能可以自动检测接头类型**，不需要你手动指定。

⚠️ **踩坑预警：手动指定了错误的接头序列**
> 有些教程会让你手动用 `-a AGATCGGAAGAGC...` 指定 TruSeq 接头——这对 ATAC-seq 是错的。如果你发现清洗后的 FastQC 报告仍然显示 adapter contamination，很可能就是指定了错误的接头。用 `--detect_adapter_for_pe` 让 `fastp` 自动识别就好。

### ATAC-seq 的 FastQC 报告怎么看？

FastQC 有 11 个检测模块。对于 ATAC-seq 数据，你需要特别关注以下几个，以及一些**预期会"报警"但其实没有问题**的模块：

| FastQC 模块 | ATAC-seq 预期表现 | 说明 |
|-------------|------------------|------|
| **Per base sequence quality** | ✅ 应该 PASS | 碱基质量应 >Q20，最好 >Q30 |
| **Per sequence quality scores** | ✅ 应该 PASS | 大部分 reads 的平均质量应 >Q30 |
| **Per base sequence content** | ⚠️ 通常 WARN/FAIL | Tn5 有轻微的序列偏好性，前 ~10 bp 碱基比例不均匀——**这是正常的** |
| **Sequence duplication levels** | ⚠️ 通常 WARN | PCR 扩增导致的重复——我们会在第 5 篇处理 |
| **Overrepresented sequences** | ⚠️ 可能 WARN | 可能出现 Nextera 接头序列 |
| **Adapter Content** | ⚠️ 通常 WARN/FAIL | **Nextera 接头污染是 ATAC-seq 的常见特征**——小片段文库中尤其明显 |
| **Per base N content** | ✅ 应该 PASS | N 碱基应极少 |
| **GC content** | ⚠️ 可能 WARN | 开放染色质区域的 GC 含量偏高，分布可能不完全正态 |

<!-- 图 1 位置：原始数据 FastQC 报告截图，展示 Per base quality 和 Adapter Content 两个模块 -->
**图 1：GM12878_rep1_R1 原始数据 FastQC 报告关键模块。** 左图：Per base sequence quality 显示碱基质量整体优良（Q > 30）。右图：Adapter Content 模块显示 Nextera Transposase 接头在 read 尾部的占比逐渐升高（红色曲线）——这是 ATAC-seq 数据的典型特征，需要在接头去除步骤中处理。

💡 **Adapter Content 模块是关键**
> 在 ATAC-seq 中，**NFR 区域产生的片段往往 <150 bp**。如果测序读长是 PE150，那么 R1 和 R2 会"读穿"片段，读到对面的接头序列。这就是为什么 Adapter Content 模块几乎总是报警——**这完全正常，但必须在比对前去除接头**。

## 步骤 3：用 fastp 去除接头和低质量序列

### 为什么选择 fastp？

接头去除工具有很多（Trimmomatic、Cutadapt、trim_galore 等），但 `fastp` 有几个突出优势：

- **自动接头检测**：`--detect_adapter_for_pe` 可以自动识别 Nextera 接头，不需要手动指定
- **速度极快**：基于 C++ 编写，比 Trimmomatic 快 3–5 倍
- **一体化 QC**：同时输出清洗前后的质量报告（JSON + HTML），不需要再跑一次 FastQC
- **碱基校正**：`--correction` 选项利用双端 reads 的重叠区域进行碱基错误校正

### 单样本运行示例

我们先用一个样本演示 `fastp` 的用法和关键参数：

```bash
# ============================================================
# 用 fastp 对单个样本进行接头去除和质量修剪
# --detect_adapter_for_pe 自动识别 Nextera 接头
# --correction 利用 PE overlap 校正碱基错误
# ============================================================
cd ~/atacseq_tutorial

sample="GM12878_rep1"

fastp \
    -i raw_data/${sample}_R1.fastq.gz \
    -I raw_data/${sample}_R2.fastq.gz \
    -o clean_data/${sample}_R1_clean.fastq.gz \
    -O clean_data/${sample}_R2_clean.fastq.gz \
    --detect_adapter_for_pe \
    --correction \
    --thread 8 \
    --qualified_quality_phred 20 \
    --length_required 35 \
    -j logs/${sample}_fastp.json \
    -h logs/${sample}_fastp.html
```

3. **17.77% 的 reads 有接头修剪**——这在 ATAC-seq 中是正常的，反映了 NFR 短片段的比例
4. **Insert size peak 在 168 bp**——接近核小体周期，符合预期
5. **重复率 22.15%**——PCR 重复在合理范围内，我们会在第 5 篇用 `picard` 精确标记和去除

⚠️ **踩坑预警：接头修剪比例过低**
> 如果你发现 `reads with adapter trimmed` 的比例低于 5%，而 FastQC 报告明明显示有接头污染——很可能是自动检测失败了。这种情况下可以手动指定 Nextera 接头序列：
> ```bash
> fastp --adapter_sequence CTGTCTCTTATACACATCT \
>       --adapter_sequence_r2 CTGTCTCTTATACACATCT ...
> ```

### 参数说明

| 参数 | 值 | 含义 |
|------|-----|------|
| `--detect_adapter_for_pe` | — | 自动检测双端测序的接头序列 |
| `--correction` | — | 利用 R1/R2 重叠区域校正碱基错误 |
| `--qualified_quality_phred` | 20 | 质量低于 Q20 的碱基被视为"不合格" |
| `--length_required` | 35 | 修剪后长度 <35 bp 的 reads 被丢弃 |
| `--thread` | 8 | 使用 8 个线程 |
| `-j` / `-h` | — | 输出 JSON 和 HTML 格式的质控报告 |

### 批量处理所有样本

```bash
# ============================================================
# 批量运行 fastp，处理所有 4 个样本
# 使用循环读取 sample_info.tsv 中的样本 ID
# ============================================================
cd ~/atacseq_tutorial

for sample in GM12878_rep1 GM12878_rep2 K562_rep1 K562_rep2; do
    echo "=========================================="
    echo "Processing: ${sample}"
    echo "=========================================="

    fastp \
        -i raw_data/${sample}_R1.fastq.gz \
        -I raw_data/${sample}_R2.fastq.gz \
        -o clean_data/${sample}_R1_clean.fastq.gz \
        -O clean_data/${sample}_R2_clean.fastq.gz \
        --detect_adapter_for_pe \
        --correction \
        --thread 8 \
        --qualified_quality_phred 20 \
        --length_required 35 \
        -j logs/${sample}_fastp.json \
        -h logs/${sample}_fastp.html \
        2>&1 | tee logs/${sample}_fastp.log

    echo ""
done
```

**清洗前后的对比是验证接头去除是否成功的最佳方式。**

