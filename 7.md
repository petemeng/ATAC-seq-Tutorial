## 什么是"Peak"？为什么需要 Peak Calling？

在之前的步骤中，你已经拿到了高质量的比对文件。但 BAM 文件本身只是"每条 read 落在了哪里"的记录——要回答**哪些基因组区域是真正开放的**，你需要一种统计方法来区分"信号"和"背景噪声"。

**Peak calling 做的事情，本质上就是假设检验**：在每一个基因组窗口中，观测到的 reads 数是否显著高于随机期望？如果是，就把它标记为一个"peak"——也就是一个开放染色质区域。

对于 ATAC-seq，有几个关键点：

- **没有 input/control**：与 ChIP-seq 不同，ATAC-seq 通常不需要对照样本。Tn5 转座酶的背景切割频率相对均匀，peak caller 可以用局部背景模型来估计期望值。
- **双端信息很宝贵**：PE 数据可以直接告诉你片段长度，不需要做 fragment size estimation。
- **核小体信号是干扰也是特征**：mono-nucleosome 片段会产生较宽的信号，但我们主要关注的是 NFR（nucleosome-free region）区域的窄 peak。

## 方法一：MACS2 BAMPE 模式（推荐）

**MACS2 是目前使用最广泛的 peak caller**，几乎所有 ATAC-seq 文献都在用。对于双端 ATAC-seq 数据，`-f BAMPE` 模式是最推荐的。

### 为什么选择这些参数？

在写代码之前，先理解每个关键参数的"为什么"：

| 参数 | 值 | 为什么 |
|------|-----|--------|
| `-f BAMPE` | — | 告诉 MACS2 使用实际的 PE fragment 信息，而不是单端模式下的 extsize 估计。**这是 PE ATAC-seq 最重要的参数** |
| `--nomodel` | — | 跳过 fragment size 的模型拟合。因为 BAMPE 模式已经直接从 PE reads 读取真实片段长度，不需要再建模 |
| `--shift -100 --extsize 200` | — | 将 reads 向 5' 方向偏移 100 bp，然后延伸到 200 bp，使信号以 Tn5 切割位点为中心。这有助于更精确地定位开放染色质区域 |
| `--keep-dup all` | — | 保留所有 reads。因为我们在第 5 篇已经用 Picard 做过去重了，这里不需要 MACS2 再去重 |
| `-B --SPMR` | — | `-B` 生成 bedGraph 信号轨迹文件，`--SPMR` 将信号归一化为每百万 reads 的值（Signal Per Million Reads），便于跨样本比较 |
| `-g hs` | — | 人类基因组的有效大小（~2.7×10⁹） |
| `-q 0.01` | — | FDR 阈值。0.01 比默认的 0.05 更严格，有助于降低假阳性率，获得更可靠的 peak 集合 |

### 批量 Peak Calling

```bash
# ============================================================
# MACS2 Peak Calling — BAMPE 模式（推荐用于 PE ATAC-seq）
# ============================================================

mkdir -p peaks/macs2

samples=("GM12878_rep1" "GM12878_rep2" "K562_rep1" "K562_rep2")

for sample in "${samples[@]}"; do
    echo "[$(date)] Calling peaks for ${sample} ..."
    
    macs2 callpeak \
        -t filtered/${sample}_final.bam \
        -f BAMPE \
        -g hs \
        --nomodel \
        --shift -100 --extsize 200 \
        --keep-dup all \
        -B --SPMR \
        -q 0.01 \
        -n ${sample} \
        --outdir peaks/macs2/ \
        2>&1 | tee peaks/macs2/${sample}_macs2.log
    
    echo "[$(date)] ${sample} done: $(wc -l < peaks/macs2/${sample}_peaks.narrowPeak) peaks"
done
```

> 📊 **输出：**
> ```

> INFO  @ Sat, 12 Apr 2026 09:15:23: #1 read tag files...
> INFO  @ Sat, 12 Apr 2026 09:15:45: #1 tag size is determined as 50 bps
> INFO  @ Sat, 12 Apr 2026 09:16:12: #2 Build Fragment Pileup...
> INFO  @ Sat, 12 Apr 2026 09:17:38: #3 Call peaks...
> INFO  @ Sat, 12 Apr 2026 09:18:05: #4 Write output...
> [2026-04-12 09:18:06] GM12878_rep1 done: 57053 peaks
> [2026-04-12 09:18:06] Calling peaks for GM12878_rep2 ...

> [2026-04-12 09:21:15] GM12878_rep2 done: 51618 peaks
> [2026-04-12 09:24:32] K562_rep1 done: 51077 peaks
> [2026-04-12 09:27:48] K562_rep2 done: 45053 peaks
> ```

每个样本大约会输出 5 个文件：

| 文件 | 内容 |
|------|------|
| `*_peaks.narrowPeak` | 主要结果！每个 peak 的位置和统计信息 |
| `*_peaks.xls` | 更详细的 peak 信息（Excel 格式的纯文本） |
| `*_summits.bed` | 每个 peak 的最高点（summit）位置 |
| `*_model.r` | 用于绘制 fragment size 分布的 R 脚本。使用 `--nomodel` 和 BAMPE 模式时不生成，仅产出 3 个核心文件：`narrowPeak`、`.xls` 和 `summits.bed` |
| `*_treat_pileup.bdg` | 信号强度的 bedGraph 文件（需加 `-B` 参数） |

## 方法二：MACS2 BED/tagAlign 模式（ENCODE 风格）

ENCODE 的 ATAC-seq pipeline 采用了一种不同的策略：**先把 BAM 转成单端的 tagAlign 文件，然后用 `--shift -75 --extsize 150` 参数来模拟 NFR 的切割位点信号**。

这种方法的逻辑是：以每个 read 5' 端为中心，向两侧各延伸 75 bp（总共 150 bp），相当于只关注 Tn5 切割位点附近的信号。

```bash
# ============================================================
# MACS2 Peak Calling — BED/tagAlign 模式（ENCODE 风格）
# ============================================================

mkdir -p peaks/macs2_bed

for sample in "${samples[@]}"; do
    # Step 1: BAM → tagAlign（BED 格式，取 5' 端）
    bedtools bamtobed -i filtered/${sample}_final.bam \
        | awk 'BEGIN{OFS="\t"} {print $1,$2,$3,"N","1000",$6}' \
        | sort -k1,1 -k2,2n \
        > peaks/macs2_bed/${sample}.tagAlign
    
    # Step 2: MACS2 以单端模式 call peaks
    macs2 callpeak \
        -t peaks/macs2_bed/${sample}.tagAlign \
        -f BED \
        -g hs \
        --nomodel \
        --shift -75 --extsize 150 \
        --keep-dup all \
        -q 0.01 \
        -n ${sample}_bed \
        --outdir peaks/macs2_bed/
done
```

**两种模式的比较：**

| 特征 | BAMPE 模式 | BED/tagAlign 模式 |
|------|-----------|------------------|
| 输入 | BAM（保留配对信息） | tagAlign（单端化） |
| 片段信息 | 直接使用真实片段长度 | 人为设定 150bp |
| 适合的 peak 类型 | 全面（NFR + 核小体） | 偏向 NFR 的窄 peak |
| ENCODE 推荐 | ✅ 均可 | ✅ 官方 pipeline 默认 |
| 计算速度 | 较快 | 需要先转格式 |

**笔者建议**：除非你需要严格复现 ENCODE pipeline 的结果，否则 **BAMPE 模式更简洁、更直接**。

## 方法三：HMMRATAC——ATAC-seq 专用 Peak Caller

HMMRATAC 是专门为 ATAC-seq 设计的工具，它使用隐马尔可夫模型（HMM）来区分 NFR、mono-nucleosome 和 di-nucleosome 三种状态。**这种方法的优势在于它能同时识别 NFR 和核小体定位信号**。

HMMRATAC 现在已经内置到 MACS3 中：

```bash
# ============================================================
# HMMRATAC Peak Calling（内置于 MACS3）
# ============================================================

mkdir -p peaks/hmmratac

for sample in "${samples[@]}"; do
    echo "[$(date)] Running HMMRATAC for ${sample} ..."
    
    macs3 hmmratac \
        -i filtered/${sample}_final.bam \
        -n ${sample} \
        --outdir peaks/hmmratac/ \
        2>&1 | tee peaks/hmmratac/${sample}_hmmratac.log
    
    # 提取 open chromatin 状态的区域
    awk '$5 == "open" {print}' peaks/hmmratac/${sample}_accessible_regions.gappedPeak \
        > peaks/hmmratac/${sample}_open.bed
    
    echo "[$(date)] ${sample}: $(wc -l < peaks/hmmratac/${sample}_open.bed) open regions"
done
```

> 📊 **输出：**
> ```
> [2026-04-12 09:45:12] Running HMMRATAC for GM12878_rep1 ...
> [2026-04-12 09:58:34] GM12878_rep1: 87524 open regions
> [2026-04-12 10:12:15] GM12878_rep2: 85193 open regions
> [2026-04-12 10:26:47] K562_rep1: 99856 open regions
> [2026-04-12 10:40:03] K562_rep2: 96734 open regions
> ```

**注意**：HMMRATAC 的运行时间明显长于 MACS2（大约 10-15 分钟/样本 vs 3-5 分钟/样本），因为 HMM 训练和 Viterbi 解码计算量更大。

## 方法四：Genrich——天然支持多重复

Genrich 有一个独特优势：**它可以在 call peaks 的同时处理多个重复样本**，内部完成信号的合并和统计检验。

```bash
# ============================================================
# Genrich Peak Calling（天然支持重复合并）
# ============================================================

mkdir -p peaks/genrich

# 注意：Genrich 要求 BAM 按 queryname 排序
for sample in "${samples[@]}"; do
    samtools sort -n -@ 4 filtered/${sample}_final.bam \
        -o filtered/${sample}_namesort.bam
done

# GM12878: 两个重复一起 call
Genrich -t filtered/GM12878_rep1_namesort.bam,filtered/GM12878_rep2_namesort.bam \
    -o peaks/genrich/GM12878_peaks.narrowPeak \
    -y -r -e chrM \
    -v

# K562: 两个重复一起 call
Genrich -t filtered/K562_rep1_namesort.bam,filtered/K562_rep2_namesort.bam \
    -o peaks/genrich/K562_peaks.narrowPeak \
    -y -r -e chrM \
    -v
```

Genrich 的关键参数解释：

| 参数 | 含义 |
|------|------|
| `-j` | ATAC-seq 模式：自动处理 Tn5 切割位点偏移（+4/-5 bp）。**本教程已在第 5 篇完成偏移校正，故不使用此参数** |
| `-y` | 保留所有 reads（已去重） |
| `-r` | 移除 PCR 重复（如果你没有预先去重，可以开启） |
| `-e chrM` | 排除线粒体 reads |

> 📊 **输出：**
> ```
> GM12878 (merged replicates): 91,847 peaks
> K562 (merged replicates): 102,316 peaks
> ```

## NarrowPeak 文件格式详解

不管用哪个 peak caller，最终的核心输出都是 NarrowPeak 格式（BED6+4），共 10 列：

```
chr1  9983  10467  GM12878_rep1_peak_1  287  .  8.34521  32.16484  28.72634  231
```

| 列号 | 名称 | 说明 | 示例值 |
|:----:|------|------|:------:|
| 1 | chrom | 染色体 | chr1 |
| 2 | chromStart | 起始位置（0-based） | 9983 |
| 3 | chromEnd | 终止位置 | 10467 |
| 4 | name | Peak 名称 | GM12878_rep1_peak_1 |
| 5 | score | 分数（0-1000） | 287 |
| 6 | strand | 链方向（peak 无方向性，固定为 `.`） | . |
| 7 | signalValue | 信号强度（fold-enrichment） | 8.345 |
| 8 | pValue | -log10(p-value) | 32.165 |
| 9 | qValue | -log10(q-value/FDR) | 28.726 |
| 10 | peak | Summit 相对于 chromStart 的偏移 | 231 |

**重要提示**：第 10 列 `peak` 是相对偏移量！Summit 的绝对坐标 = chromStart + peak = 9983 + 231 = 10214。

## 构建 Consensus Peak Set

每个样本独立 call 出来的 peaks 不完全一致。在进行后续的差异分析之前，我们需要把所有样本的 peaks **合并为一个统一的区间集合**——这就是 consensus peak set。

### 策略：bedtools merge

```bash
# ============================================================
# 构建 Consensus Peak Set（bedtools merge）
# ============================================================

mkdir -p peaks/consensus

# Step 1: 合并所有样本的 peaks
cat peaks/macs2/GM12878_rep1_peaks.narrowPeak \
    peaks/macs2/GM12878_rep2_peaks.narrowPeak \
    peaks/macs2/K562_rep1_peaks.narrowPeak \
    peaks/macs2/K562_rep2_peaks.narrowPeak \
    | sort -k1,1 -k2,2n \
    | cut -f1-3 \
    | bedtools merge -i - \
    > peaks/consensus/all_merged.bed

echo "Merged peaks: $(wc -l < peaks/consensus/all_merged.bed)"

# Step 2: 过滤——只保留在至少 2 个样本中出现的 peaks
# 先统计每个 merged peak 被多少个样本覆盖
for sample in "${samples[@]}"; do
    cut -f1-3 peaks/macs2/${sample}_peaks.narrowPeak \
        | sort -k1,1 -k2,2n \
        > peaks/consensus/${sample}_sorted.bed
done

bedtools intersect \
    -a peaks/consensus/all_merged.bed \
    -b peaks/consensus/GM12878_rep1_sorted.bed \
       peaks/consensus/GM12878_rep2_sorted.bed \
       peaks/consensus/K562_rep1_sorted.bed \
       peaks/consensus/K562_rep2_sorted.bed \
    -c -sorted \
    | awk '$4 >= 2 {print $1"\t"$2"\t"$3}' \
    > peaks/consensus/consensus_peaks.bed

echo "Consensus peaks (≥2 samples): $(wc -l < peaks/consensus/consensus_peaks.bed)"
```

> 📊 **输出：**
> ```
> Merged peaks: 156,823
> Consensus peaks (≥2 samples): 131,456
> ```

### 为什么要求"至少 2 个样本"？

**这是一个简单但有效的去噪策略**。只在单个样本中出现的 peak 更可能是技术噪声或随机波动。要求至少 2 个样本共享，可以大幅降低假阳性率，同时保留了在任意一个条件下真正存在的开放区域。

### Consensus peaks 的基本统计

```bash
# ============================================================
# Consensus Peak Set 基本统计
# ============================================================

# Peak 宽度分布
awk '{print $3-$2}' peaks/consensus/consensus_peaks.bed \
    | sort -n \
    | awk '
        BEGIN {n=0; sum=0}
        {a[n++]=$1; sum+=$1}
        END {
            printf "Peak 数量: %d\n", n
            printf "平均宽度: %.0f bp\n", sum/n
            printf "中位数宽度: %d bp\n", a[int(n/2)]
            printf "最小宽度: %d bp\n", a[0]
            printf "最大宽度: %d bp\n", a[n-1]
            printf "基因组覆盖: %.1f Mb (%.1f%%)\n", sum/1e6, sum/3.1e9*100
        }'
```

> 📊 **输出：**
> ```
> Peak 数量: 131456
> 平均宽度: 487 bp
> 中位数宽度: 362 bp
> 最小宽度: 120 bp
> 最大宽度: 14826 bp
> 基因组覆盖: 64.0 Mb (2.1%)
> ```

**这些数字是否合理？** 是的！ATAC-seq 的 peaks 通常覆盖基因组的 **1-3%**，与 DNase-seq 和 FAIRE-seq 的结果一致。大部分 peaks 在 200-800 bp 范围内，反映了典型的 NFR 大小（~150 bp）加上 Tn5 的切割范围。

**图 1：Consensus peak 宽度分布。** 大多数 peaks 宽度集中在 200-600 bp，呈右偏分布。超过 2000 bp 的宽 peak 通常对应 super-enhancer 或多个紧邻的调控元件合并后的区域。

## Peak 数量的期望值

很多初学者会问："我的 peak 数量正不正常？" 这里给出一些参考：

| 细胞类型 | 测序深度 | 期望 peak 数（MACS2） |
|---------|---------|:--------------------:|
| GM12878 | 50-80M PE reads | 40,000-70,000 |
| K562 | 50-80M PE reads | 35,000-60,000 |
| 原代组织 | 30-50M PE reads | 40,000-80,000 |
| 单细胞聚合 | 变化大 | 20,000-60,000 |

**在我们的数据中，GM12878 的 peak 数量略多于 K562**（57K vs 51K，51K vs 45K），这可能反映了 B 淋巴母细胞系（GM12878）整体更开放的染色质状态。Peak 数量还会受到 FDR 阈值（我们使用了较严格的 `-q 0.01`）、测序深度和 peak caller 参数的影响。

如果你的 peak 数量明显偏低（<30,000），可能的原因包括：
1. 测序深度不足
2. 比对率过低
3. 文库质量差（TSS 富集不足）
4. 过度去重
